HQ Knowledge — one unified browser

Version 1 · 2026-06-22 · Owner: Sonja Streich · Status: DRAFT — OQs open

North star: retire Notion. The Databases primitive (B1/B2) is in; this spec reconciles the three "document" systems that grew up separately — Pages, Docs, and Databases — into a single browser where every item is a first-class, filterable, editable node.

1 · The problem — three islands, two muddled axes

The Knowledge hub today carries three different "document" systems that were each built on their own and never reconciled. They store data differently, allow editing in different (or no) places, and respond to the toolbar filters inconsistently. The B2 Notion import made this visible: the Metrics & Calculations database came in as a real database, but nothing connects it to the Semantic Layer doc it lived inside in Notion.

Docs (+ Ideas / Chats / Notes) knowledge_entries, kind='doc' — the imported Notion HTML (Semantic Layer, the 21 migrated docs) lives here No — uploaded HTML renders in a read-only "Original" iframe (EntryDetail ~L524). Editing the body never touches the rendered HTML. ✅ Kind + Entity + Search drive listEntries()

Vault knowledge_entries, access='vault' File only (no body editing) ❌ filters disabled on this tab

Two axes are conflated into one toolbar

The current toolbar mixes two different questions onto controls that sit side by side:

  • "What content type?" — Ideas / Docs / Chats / Notes (the kind filter) … and also Pages / Databases / Vault (which are content types too, but live on the view toggle).
  • "How should it be displayed?" — Cards / List / Insights (genuine display modes, on the same toggle as Pages/Databases).

Because Pages and Databases are content-type selectors masquerading as view modes, and the Kind/Entity filters only feed the entry-list views, the filters go dead the moment you're on the Pages or Databases tab — they aren't "relative to" that view because the view is itself a competing content selector. (Concretely: handleKindChange only re-runs listEntries(); WorkspaceView / DatabasesView never read kind or entity.)

Bonus inconsistency: listEntries' "All" does .neq('kind','critique') — so it includes workspace pages. A page therefore shows up twice: once as a card in Cards/List "All", and again in the Pages tree. The same item has two homes with two behaviours.

2 · The decision (locked 2026-06-22)

Full unify into one browser. Collapse Pages, Docs, and Databases into a single, filterable browser where every item — a page, a doc, a database, an idea, a note, a chat, a vault file — is a first-class node with consistent filtering, a consistent way to open it, and editing appropriate to its type. One place, two clean axes (type · scope), one display layer. (Chosen over "link-only" and "promote-docs-to-pages" — both were partial. This is the true end state.)

3 · Target model — the unified browser

Three orthogonal controls, each doing exactly one job, every one applying to everything:

AxisControlApplies to
Type (what)All · Pages · Docs · Databases · Ideas · Notes · Chats · Vault — a single segmented filterevery node, from one index
Scope (which)Entity (the 7) · Search — text + type-aheadevery node, uniformly
Display (how)List · Cards · Tree — orthogonal, remembered per sessionwhere meaningful (Tree shows nesting; Insights stays a separate analytical view)

"Pages", "Databases", and "Vault" stop being tabs and become type filters sitting next to Docs/Ideas/Notes/Chats. Pick a type (or All) → choose a display → filter by entity/search. Every result is a node you can open; opening it lands on a detail view whose editor matches its type (Markdown editor for pages, table editor for databases, Original+Edit for HTML docs).

Nesting & containment — the Notion shape

A unified browser needs one containment graph. Pages already have parent_id. The model extends that so a node can contain other nodes: a page can hold sub-pages, an embedded database, and attached docs. The Tree display renders this graph; the flat List/Cards displays render the filtered set. This is what finally lets the Semantic Layer page contain its two databases again (see §6).

4 · Architecture — the "node" abstraction

Two stores stay (knowledge_entries and hq_databases) — no destructive merge. They unify at a thin presentation/index layer: a KnowledgeNode view model that both stores project into. 

type KnowledgeNode = {
  id: string
  source: 'entry' | 'database'      // which store it came from
  type:  'page' | 'doc' | 'idea' | 'note' | 'chat' | 'database' | 'vault'
  title: string
  icon:  string | null
  entities: string[]                // unified entity tags (junction)
  parent_id: string | null          // containment graph (pages today; nodes tomorrow)
  updated_at: string
  // editing/open affordance derived from `type`
}
  • Index: a single server reader (listNodes({ type, entity, query, display })) unions the two stores into KnowledgeNode[], applying entity/search uniformly. Replaces the split between listEntries / listWorkspaceTree / listDatabases at the UI boundary (the underlying readers stay).
  • Containment: introduce a node-level parent link so a database (a database node) can be parented under a page. Cleanest is a small knowledge_node_links / reuse of the existing graph rather than putting a parent_id on hq_databases. → OQ-1.
  • Embedding (the inline table): knowledge_db_embeds(host_entry_id, database_id, view_config jsonb) — a page renders an embedded database (optionally a filtered/sorted view) inline. This is the deferred OQ-3 from the Databases spec, pulled in here.
  • Linking (DB ↔ doc, both directions): reuse the existing knowledge_links substrate (already powers doc↔project attached + mentions), extended to allow a database as a link target. Gives the "appears on / related to" backlink between the Metrics DB and the Semantic Layer doc with no new mechanism.
  • Editing parity: (a) an Edit mode on the Original tab for HTML docs (today a read-only iframe — no edit, no download); (b) database row/column editing (B3 core); (c) keep the Markdown editor for pages. After this, every node type is editable in place.
  • Promotion (optional, not bulk): a "Convert to Page" action turns an import-doc that was a database-container in Notion into an editable page that embeds its databases, keeping the imported HTML as a read-only Original snapshot/version. Used surgically (Semantic Layer, Gate In/Out), not across all 21 docs.
Why a presentation-layer union, not one mega-table: a destructive merge of knowledge_entries and hq_databases would be high-risk and gains little — they have genuinely different shapes (prose vs typed records). Unifying at the index + containment + link layers gives the single-browser experience without a dangerous migration, and keeps RLS exactly as it is (org-scoped on both stores).

6 · Worked example — the Semantic Layer

This is the case that prompted the spec. After the unification:

  • The Semantic Layer doc becomes (via "Convert to Page") an editable Page with its intro prose.
  • It embeds the two imported databases inline — 🍎 UI App Sections and ♎ Metrics & Calculations — rebuilding the exact Notion structure (a page that contained those databases).
  • The imported HTML is retained as the read-only Original snapshot/version, so nothing from the migration is lost.
  • Each database shows an "appears on → Semantic Layer" backlink, so from the Metrics DB you can navigate to the page it belongs to.
  • The Metrics DB's App Sections relation (today raw Notion ids) resolves to the App Sections rows once both DBs are linked — the cross-DB relation resolution noted in B2.

8 · Open questions — to lock before U1

OQ-1 · Containment mechanism for non-page nodes

How does a database (or doc) get parented under a page? (a) a generic knowledge_node_links(parent_id, child_id, child_source) graph table; (b) add a nullable parent_entry_id to hq_databases; (c) lean on the existing knowledge_links with a contains relation. Recommendation: (a) — one graph for all node types, future-proof for embedding docs-in-docs too.

OQ-2 · Index strategy

Union the two stores in app code per request (simple, always fresh) vs a materialized knowledge_nodes index view/table kept in sync (faster sort/paginate across stores, more machinery). Recommendation: app-code union for U1 (volumes are small); revisit if cross-store sort/pagination gets slow.

OQ-3 · Does "Vault" stay a type, or its own gate?

Vault entries are owner-only (Tier-2). Fold them in as a type filter in the same browser (consistent), or keep a distinct locked surface? Recommendation: type filter, but keep the owner-only RLS + the amber treatment so it reads as privileged.

OQ-4 · Promotion default — copy vs convert-in-place

When promoting an import-doc to a Page: convert the same entry's kind doc→workspace (keeps id, links, history; Original tab holds the old HTML) vs create a new page and archive the doc. Recommendation: convert in place — preserves every backlink and the id the import set up.

OQ-5 · How far does "Tree" go in U1?

Ship the unified flat List/Cards in U1 and add the containment Tree in U3 (when embeds/parenting exist), or build a basic Tree in U1? Recommendation: List/Cards in U1; Tree lands with containment in U3 so it has real nesting to show.
