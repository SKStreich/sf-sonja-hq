Migration shape

-- migration: 20260526000001_canonical_role_spine.sql
BEGIN;

-- PG 17 supports both operations inside a transaction.
ALTER TYPE user_role RENAME VALUE 'owner' TO 'platform_owner';
ALTER TYPE user_role RENAME VALUE 'admin' TO 'org_admin';
ALTER TYPE user_role ADD VALUE 'supervisor' AFTER 'org_admin';

-- RLS policies that hard-coded old names (must drop+recreate)
DROP POLICY "integrations_admin_only" ON integrations;
CREATE POLICY "integrations_admin_only" ON integrations FOR ALL
  USING (
    org_id = (SELECT org_id FROM user_profiles WHERE id = auth.uid())
    AND EXISTS (
      SELECT 1 FROM user_profiles
      WHERE id = auth.uid()
        AND role IN ('platform_owner', 'org_admin')
    )
  );

-- Same shape for the 2 policies in 20260425…contacts_and_share_forwarding.sql

COMMIT;
Migration gotcha — verify in dev first
Per project_sonja_hq_tech.md: "ALTER TYPE … ADD VALUE must commit before the new value can be used in the same migration." This migration only adds 'supervisor' — no policy or default uses it yet (capability gates added in code, not migration). Should run in one TX on PG 17. If migration fails, split ADD VALUE into a separate apply_migration call.
