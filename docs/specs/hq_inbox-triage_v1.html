HQ Inbox & Triage

Version 1 · 2026-06-24 · Owner: Sonja Streich · Status: DRAFT — 3 decisions locked, 3 open

North star: HQ is the master of truth. Today, anything captured quickly (Siri, the agent, the capture API) drops straight into the main feed with a guessed entity and is easy to lose. This spec adds a lightweight triage queue — one surface where un-filed items wait until you give them a home — without disturbing the lifecycle (active/archived) and idea-maturity workflows that already exist.

1 · The problem

HQ now ingests from several low-friction paths — the capture API (src/app/api/capture/actions.ts), Siri (src/app/api/siri/route.ts), and the HQ agent. These create knowledge_entries immediately, with whatever entity was guessed, and the row joins the main feed. There is no "this still needs filing" state, so quick captures are indistinguishable from deliberate, well-classified docs. When the first bulk importer lands (OneDrive folders, a Notion archive, Granola transcripts), that problem scales: hundreds of rows would pour into the feed with no holding area.

This is concept #3 from the preserved-architecture note (HQ Knowledge ae15bcf5). Concepts #1 (per-entry staleness) and #2 (Areas) are separate, later slices; this spec deliberately scopes to inbox-triage only, but is designed so staleness and Areas slot in cleanly afterward.

The original 2026-05 proposal predates the Unified Knowledge Browser and assumed a fresh status enum. Three things in current HQ reshape it:

3 · Locked decisions

  • D1 — Two-state triage. A new column triage_status with exactly inbox / filed. "needs_review" belongs to the later staleness slice; "archived" stays in status. LOCKED
  • D2 — Auto-route the quick-capture paths. The capture API, Siri, and the agent create entries as triage_status='inbox' with no forced entity. Deliberate manual creates and entity-tagged uploads are born filed. Future bulk importers also land in inbox. LOCKED
  • D3 — Complement the Ideas inbox. No migration or retirement of the existing idea-maturity surface. LOCKED

4 · Data model

-- migration: knowledge_entries.triage_status
ALTER TABLE knowledge_entries
  ADD COLUMN triage_status TEXT NOT NULL DEFAULT 'filed'
  CHECK (triage_status IN ('inbox','filed'));

-- Partial index for the inbox queue read (small, hot subset).
CREATE INDEX knowledge_entries_inbox_idx
  ON knowledge_entries (org_id, created_at DESC)
  WHERE triage_status = 'inbox';
  • Default filed so the 63 existing rows + every deliberate create stay out of the inbox; only the auto-routed paths (D2) set inbox.
  • No new RLS — the column rides existing knowledge_entries policies.
  • Entity stays multi-entity via the junction; an inbox item simply has zero junction rows until triaged (the "no forced entity" half of D2). The Inbox view shows "no entity" plainly.

5 · UI

  • 📥 Inbox type filter in KnowledgeHub's Type row (peer of All·Pages·…·Vault). Selecting it lists triage_status='inbox' nodes via listNodes({triage:'inbox'}), newest-first.
  • Inline triage on each inbox node in NodeView: an entity multi-select (reuses EntityMultiSelect) + a "File" button that writes the entity junction and flips triage_status='filed' in one action. Optional "File & open".
  • Dashboard chip: a "📥 N to triage" Insight on /dashboard linking into the Inbox filter — turns the queue into a visible, shrinking number (mirrors the planned staleness "needs review" chip).
  • Keyboard-light: v1 is click-driven (pick entity → File). A j/k + "f to file" keyboard pass is a fast-follow, not v1.

6 · Phased plan

PhaseScope
T1 — substrate + readMigration (triage_status + index). listNodes gains a triage filter; TYPE_FILTERS gains 📥 Inbox; count helper. No writes yet — existing data all filed, so the inbox renders empty.
T2 — auto-route + triage actionCapture API / Siri / agent set triage_status='inbox' + skip entity assignment. New server action fileEntry(id, entities) (set junction + flip to filed). Inline triage UI in NodeView + dashboard chip.
T3 — (later) lineage + bulk importWhen the first bulk importer is built: add external_source / external_ref / external_last_edited_at, route imports to inbox, and make re-runs non-destructive (refresh external fields only; never overwrite human triage). Out of scope for this spec.

7 · Open questions

OQ-1 · Does the main feed hide inbox items?

When 📥 Inbox is its own filter, should the default All view exclude triage_status='inbox' (so the feed stays "filed, trustworthy" content and the inbox is the only place unfiled items appear), or include them with an "unfiled" badge? Recommendation: exclude from All — the whole point is to keep un-triaged captures out of the main feed.

OQ-2 · Can an item be filed with no entity?

Is picking ≥1 entity required to file, or can you "file as-is" (entity stays empty, just leaves the queue)? Recommendation: require ≥1 entity to file — filing is "give it a home"; an entity-less filed item recreates the original problem.

OQ-3 · Does the agent still guess an entity?

Today the agent/capture may infer an entity. Under D2 they create as inbox with no entity. Should they instead pre-fill a suggested entity in the triage UI (you confirm/correct), per the standing "ingestion learns from corrections" rule? Recommendation: yes — land in inbox, but carry the AI's guess as a pre-selected suggestion you confirm, so triage is one click when the guess is right and the model still sees corrections.

8 · Why this is the right first Sprint-13 slice

It is the smallest of the three preserved concepts, it unblocks every future "move content into HQ" effort (the holding area is the precondition for any bulk importer), and it is additive + low-risk: one nullable-defaulted column, one type filter, one server action — all behind the now-required RLS + integration CI gate. Staleness (#1) and Areas (#2) layer on top using the same dashboard-chip and filter-pair patterns this slice establishes.
