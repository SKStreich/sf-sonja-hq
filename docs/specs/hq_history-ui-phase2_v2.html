Trigger semantics + actor threading (Q4)

The mechanism

Triggers read the actor from a session-local Postgres setting app.current_user_id, falling back to auth.uid() if not set. User-scoped Server Actions need no changes — auth.uid() resolves correctly. Service-role / admin-client Server Actions call a tiny helper RPC set_history_actor(uuid) at the top of their transaction; the trigger then picks up that user_id for any subsequent UPDATEs in the same transaction. 

-- Helper: callers set this once per transaction before UPDATEs they want attributed.
-- The `is_local=true` argument scopes the setting to the current transaction.
CREATE OR REPLACE FUNCTION set_history_actor(user_id UUID) RETURNS void AS $$
BEGIN
  PERFORM set_config('app.current_user_id', user_id::text, true);
END;
$$ LANGUAGE plpgsql;

CREATE OR REPLACE FUNCTION record_task_history() RETURNS TRIGGER AS $$
DECLARE
  actor UUID := COALESCE(
    NULLIF(current_setting('app.current_user_id', true), '')::uuid,
    auth.uid()
  );
BEGIN
  IF NEW.status IS DISTINCT FROM OLD.status THEN
    INSERT INTO entity_history (org_id, entity_type, entity_id, field_name,
                                previous_value, new_value, changed_by)
    VALUES (NEW.org_id, 'task', NEW.id, 'status',
            OLD.status::text, NEW.status::text, actor);
  END IF;
  -- repeat IS DISTINCT FROM block for priority, due_date, assignee_id, project_id
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER tasks_history_capture
  AFTER UPDATE ON tasks
  FOR EACH ROW EXECUTE FUNCTION record_task_history();

TypeScript wrapper

// src/lib/supabase/history-actor.ts
//
// Use when calling tasks/projects UPDATEs via a service-role / admin client
// where auth.uid() returns NULL. User-scoped clients don't need this — the
// trigger already resolves auth.uid() correctly for them.
//
// Caller is responsible for batching the set_history_actor + UPDATE in a
// single transaction (PostgREST exposes RPC + DML on the same connection
// when called sequentially through a single client instance).

export async function setHistoryActor(
  client: SupabaseClient,
  userId: string,
) {
  const { error } = await client.rpc('set_history_actor', { user_id: userId })
  if (error) throw error
}

RLS — unchanged from v1

ALTER TABLE entity_history ENABLE ROW LEVEL SECURITY;

CREATE POLICY "entity_history_read" ON entity_history
  FOR SELECT USING (
    org_id = (SELECT org_id FROM user_profiles WHERE id = auth.uid())
    AND (
      entity_type = 'project'
      OR entity_type = 'task' AND EXISTS (
        SELECT 1 FROM tasks t WHERE t.id = entity_history.entity_id
      )
    )
  );

-- Append-only. No INSERT/UPDATE/DELETE policies — only the SECURITY DEFINER
-- trigger writes; everything else is blocked.

Merged dashboard feed (Q1)

Postgres view: recent_activity_feed

A single view that UNIONs entity_history + project_updates into a normalized shape. RLS cascades through both underlying tables. Created with SECURITY INVOKER so it respects the caller's row-level access. 

CREATE VIEW recent_activity_feed
WITH (security_invoker = true) AS
SELECT
  'field_change'::text      AS activity_type,
  eh.id                  AS id,
  eh.org_id              AS org_id,
  eh.changed_at          AS occurred_at,
  eh.changed_by          AS actor_id,
  eh.entity_type::text    AS entity_type,
  eh.entity_id           AS entity_id,
  eh.field_name          AS field_name,
  eh.previous_value      AS previous_value,
  eh.new_value           AS new_value,
  NULL::text             AS update_content,
  NULL::text             AS update_subtype
FROM entity_history eh
UNION ALL
SELECT
  'project_update'::text,
  pu.id, pu.org_id, pu.created_at, pu.user_id,
  'project'::text, pu.project_id,
  NULL::text, NULL::text, NULL::text,
  pu.content, pu.update_type
FROM project_updates pu;

Server query

Cursor-based pagination: pass the previous page's last occurred_at as before_cursor. Initial load uses NOW(). Page size = 20. 

SELECT
  f.*,
  up.full_name AS actor_name,
  COALESCE(t.title, p.name)         AS entity_name,
  t.project_id                       AS task_project_id,
  pp.name                            AS task_project_name,
  asg.full_name                      AS new_assignee_name
FROM recent_activity_feed f
LEFT JOIN user_profiles up ON up.id = f.actor_id
LEFT JOIN tasks t          ON f.entity_type = 'task'    AND t.id = f.entity_id
LEFT JOIN projects p       ON f.entity_type = 'project' AND p.id = f.entity_id
LEFT JOIN projects pp      ON t.project_id = pp.id
LEFT JOIN user_profiles asg ON f.field_name = 'assignee_id' AND asg.id = f.new_value::uuid
WHERE f.occurred_at < $1   -- before_cursor
ORDER BY f.occurred_at DESC
LIMIT 21;                  -- 20 to render + 1 to detect "has more"

Server takes 21 rows, returns the first 20, and if a 21st exists exposes nextCursor = rows[19].occurred_at. "Show more" button visible iff nextCursor is set.

Filter pills

Client-side filter on the discriminator: all (default) · changes (entity_history rows) · updates (project_updates rows). Filter state lives in URL query param (?activity=changes) so it persists across reloads. Server returns all activity types; client filters by hiding rows whose flag doesn't match. Cursor stays activity-type-agnostic to keep pagination semantics simple.

Test plan

  • Trigger correctness (real DB per feedback-testing):
    • UPDATE that changes status emits exactly 1 row with correct prev/new/actor.
    • UPDATE that changes 3 tracked fields emits 3 rows, identical changed_at.
    • UPDATE touching only updated_at (no tracked field) emits 0 rows.
    • UPDATE NULL → value emits a row; UPDATE NULL → NULL emits 0 rows.
    • Actor threading: admin-client UPDATE without set_history_actor emits a row with changed_by = NULL; admin-client UPDATE with set_history_actor(X) emits a row with changed_by = X.
  • RLS isolation:
    • User in org A cannot read history rows for org B.
    • User cannot read history for a private task that isn't theirs.
    • Direct INSERT into entity_history by authenticated user is rejected.
    • View recent_activity_feed respects RLS via security_invoker = true.
  • ActivityFeed component:
    • Renders all 6 change templates with correct copy.
    • Renders project_update rows interleaved with change rows in correct chrono order.
    • Filter pills hide/show rows correctly; URL param syncs.
    • "Show more" button visible iff cursor present; click loads next page.
    • Renders "(deleted task/project)" when join misses; "(System)" when actor missing.
