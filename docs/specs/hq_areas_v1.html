HQ Areas — an Entity→Area middle layer

Version 1 · 2026-06-25 · Owner: Sonja Streich · Status: ✅ LOCKED 2026-06-25

Concept #2 of the preserved-architecture note (ae15bcf5). Staleness (#1) and Inbox & Triage (#3) are shipped; Areas is the last unbuilt concept. North star: HQ is the master of truth. Today the only structural axis is entity (a platform/personality) + flat tags. Tags multiply and impose no order; there is no "area within an entity" tier. This spec adds that tier so "where does this belong?" is one short navigation, not a search.

1 · The problem

HQ structure today is two-level: an item belongs to one or more entities (tm, cthq, sfe, sfo, sfs, sfc, personal — 7 platforms) and carries flat tags. That's too coarse on top and too unstructured below: a single entity like Streich Force Operations spans Work Orders, Invoicing, Field Ops, Comms, Platform — all jumbled together — while tags (#invoice, #stripe, …) don't group into anything navigable. There's no canonical answer to "show me everything under SF Ops → Invoicing."

Areas are that missing middle tier: a small, ordered, per-entity set of buckets that knowledge entries, projects, and tasks can be filed under. They complement tags (areas impose order; tags stay free-form) and complement entities (an area lives inside exactly one entity).

2 · Reconciliation with what already exists

  • Entities are multi-valued via junctions: knowledge_entry_entities(entry_id, entity TEXT) and project_entities(project_id, entity_id); tasks are single-entity (tasks.entity_id NOT NULL → entities.id). Areas must coexist with this, not replace it.
  • The original May-2026 proposal is obsolete on two points the concept note already flagged: it predates the multi-entity cutover (so its single area_id FK is too rigid) and predates the current 7-entity registry (its "SF Core / SF Facilities / SF Solutions" examples map onto today's sfo/sfs/sfc/cthq split). This v1 re-grounds both.
  • Areas are a filter axis, not a new kind. Like the 📥 Inbox and 🕓 Review filters shipped earlier this sprint, Area is a scope over a relationship, layered onto the existing Knowledge / Projects / Tasks surfaces — no new top-level navigation.

3 · Locked decisions

D1–D4 were chosen by Sonja on 2026-06-25; D5–D8 follow from them.

#DecisionDetail
D1An area belongs to exactly one entity areas(org_id, entity, name, slug, sort_order), unique on (org_id, entity, slug). entity is the TEXT slug (same convention as knowledge_entry_entities). Org-scoped.
D2Sonja Multi-area via junctions An item may sit in several areas. One junction per item type — knowledge_entry_areas, project_areas, task_areas — mirroring the multi-entity junction + RLS pattern. No single area_id FK.
D3Sonja All three item types in v1 Knowledge entries, projects, AND tasks get areas — phased A1 (substrate + manage) → A2 (knowledge) → A3 (projects + tasks).
D4Sonja Seed + manage UI A migration seeds a starter set for the high-traffic entities; a Settings surface lets Sonja add / rename / reorder / delete areas per entity. Sonja owns the taxonomy from day one.
D5Seed high-traffic entities only Sonja Seed tm, sfo, sfs, personal (§5). The other three (sfe, sfc, cthq) start empty and Sonja adds areas via the manage UI when needed.
D6Area picker is scoped to the item's entities (soft rule) When tagging an item, the area picker offers only areas whose entity is one of the item's current entities. This is enforced in the app, NOT a DB constraint — entities can change after filing, and we never want a membership edit to strand an existing area tag.
D7Area filter appears under a selected entity In each surface (Knowledge / Projects / Tasks) the Area filter shows when a single entity is selected, listing that entity's areas + "No area". With entity = All, the Area filter is hidden (areas across 7 entities would be noise). Selecting areas narrows with OR semantics.
D8Delete cascades the junction, never the item Junctions are ON DELETE CASCADE from both parents. Deleting an area removes its tags (items lose that area) but touches no items; deleting an item removes its area links.

4 · Data model

One new table + three junctions, all additive. Mirrors the multi-entity junction RLS pattern (20260531000001): junctions carry org_id, read visibility rides the parent, writes are org-scoped.

-- The areas catalogue (one row per area; an area lives in exactly one entity).
CREATE TABLE areas (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id      UUID NOT NULL REFERENCES orgs(id) ON DELETE CASCADE,
  entity      TEXT NOT NULL CHECK (entity = ANY (ARRAY
                ['tm','cthq','sfe','sfo','sfs','sfc','personal'])),
  name        TEXT NOT NULL,
  slug        TEXT NOT NULL,
  sort_order  INTEGER NOT NULL DEFAULT 0,
  created_by  UUID REFERENCES auth.users(id),
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE (org_id, entity, slug)
);
ALTER TABLE areas ENABLE ROW LEVEL SECURITY;
CREATE POLICY areas_all ON areas FOR ALL
  USING (org_id = get_my_org_id()) WITH CHECK (org_id = get_my_org_id());

-- One junction per item type (multi-area). PK dedupes; index serves the filter.
CREATE TABLE knowledge_entry_areas (
  entry_id UUID NOT NULL REFERENCES knowledge_entries(id) ON DELETE CASCADE,
  area_id  UUID NOT NULL REFERENCES areas(id)            ON DELETE CASCADE,
  org_id   UUID NOT NULL REFERENCES orgs(id)             ON DELETE CASCADE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  PRIMARY KEY (entry_id, area_id)
);
-- …project_areas(project_id, area_id, org_id) and task_areas(task_id, area_id, org_id)
--   follow the identical shape. Each: RLS on, read via parent subquery, write org-scoped.
CREATE INDEX idx_kea_area ON knowledge_entry_areas (org_id, area_id);

Helpers mirror src/lib/entities/multi-entity.ts: setEntryAreas / fetchEntryAreaMap / fetchEntryIdsForArea (+ project/task equivalents). A pure src/lib/areas/config.ts is unnecessary — areas are data, not a compile-time registry — but a small typed reader module is.

6 · UI

  • Manage (Settings → Areas): per-entity lists with add / rename / reorder (sort_order) / delete; delete confirms "N items will lose this area". The one place the catalogue is edited.
  • Assign: an AreaMultiSelect (sibling of EntityMultiSelect) on the Knowledge composer + EntryDetail, the project detail, and the task detail — scoped to the item's entities (D6). Area chips render on cards / rows next to the entity chips.
  • Filter: in Knowledge / Projects / Tasks, an Area sub-filter appears when one entity is selected (D7): that entity's areas + "No area", OR-narrowing. Deep-linkable (?entity=sfo&area=invoicing).

7 · Phased plan

SliceScope
A1 — substrate + manage Migration (areas + 3 junctions + RLS + indexes) · orgs-driven seed (§5) · area reader helpers · Settings → Areas management UI (add/rename/reorder/delete). No item assignment yet. Unit-test the pure helpers; integration lane replays the seed clean.
A2 — knowledge AreaMultiSelect on composer + EntryDetail · area chips on node cards · the Knowledge hub Entity→Area filter (D7) · listNodes/listEntries area scope.
A3 — projects + tasks Area assignment + chips + list filters on the Projects and Tasks surfaces, reusing A2's component and reader patterns.

9 · Why this is the right next slice

It's the last unbuilt concept from ae15bcf5, and it completes the "HQ as source of truth" structural story: entity (where it lives) → area (what part of it) → the item. It reuses patterns already proven this sprint — the multi-entity junction + RLS shape, the filter-as-scope toolbar idiom, the orgs-driven replay-clean seed — so the risk is in the taxonomy, not the mechanics. And it directly answers the cross-project audit's "where do documents live?" question with one canonical navigation.
