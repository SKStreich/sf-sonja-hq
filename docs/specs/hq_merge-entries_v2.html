Locked decisions

OQ1
→ NEW ENTRY
Where does the merged content land?
A brand-new knowledge_entries row. Both/all sources are archived and linked to it (merged_into). Cleanest provenance — the result is clearly "the merge," sources stay intact and recoverable.
OQ2
→ MULTI-SELECT BAR
How do you start a merge?
Checkboxes in Cards/List → a "Merge (N)" bar appears when 2+ are selected. Natural for "these look similar"; works across the whole hub. (Built in PR 2.)
OQ3
→ ANYTHING incl. WORKSPACE
What's eligible to merge?
Any active standard entry of any kind, including workspace pages. Vault is the one hard exclusion (Claude never reads vault). When a workspace-page source has children, those children are re-parented to the merged result so no page subtree is orphaned — see "Workspace re-parenting" below.
OQ4
→ UNION
Metadata of the merged result
Entities = union of all sources' entities; attached projects = union of all sources' attached projects; tags = union of all sources' tags. Title + kind + type-hint are AI-suggested, all editable in review. Loses nothing; you trim during review.
OQ5
→ FLAG CONFLICTS
Conflict handling in the draft
When sources contradict, Claude keeps both sides and appends a trailing ## Source notes & conflicts section listing the contradictions for you to resolve in review. Never silently drops a claim.
OQ6
→ SOFT UNDO
Can a merge be undone?
Sources stay status='archived' and linked. "Unmerge" = restore the archived sources to active + delete the merged entry (manual, low-frequency). All the data to recover is still there. No automatic snapshot table in v1.
OQ7
→ SONNET + ~40k CAP
Model + size guardrail
claude-sonnet-4-6 (same model as classify()). Combined source bodies are capped at ~40,000 chars; if exceeded, bodies are truncated and the draft is flagged truncated:true so the review UI can warn. If no API key is configured, the server falls back to a deterministic concatenation (sections per source) so the feature still works.

How "lossless" is enforced

The merge prompt instructs Claude to (a) include every distinct fact / point / section from each source, (b) de-duplicate overlapping content, and (c) when two sources contradict, keep both and surface the conflict in a trailing ## Source notes & conflicts section rather than silently picking one. You resolve conflicts during review. Vault entries are never read by Claude, so they cannot be merge sources.

Workspace re-parenting (OQ3 = anything)

Because workspace pages are eligible sources and a workspace page can own a child subtree (knowledge_entries.parent_id), merging them needs an explicit rule so children aren't orphaned under an archived page:

  • If any source is a workspace page, the merged result's kind is forced to workspace (overriding the AI suggestion) so it can hold a subtree and render in the workspace tree.
  • Each workspace source's direct children are re-parented to the merged result (UPDATE knowledge_entries SET parent_id = <merged> WHERE parent_id = <source>), org-scoped.
  • The merged result inherits a parent_id only if all workspace sources share one common parent; otherwise it becomes a top-level page (parent_id = null).
  • Sources are then archived. Their own parent_id is left as-is (harmless; they're archived).
