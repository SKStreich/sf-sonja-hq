HQ Daily Journal — diary + auto context

Version 1 · 2026-07-13 · Owner: Sonja Streich · Status: ✅ LOCKED 2026-07-13

A new feature (not from the ae15bcf5 preserved-architecture set). North star: a daily page Sonja writes or dictates on, with that day's HQ activity surfaced alongside as memory and prompts. The journal is a hybrid: the left side is a diary (hers, freeform); the right side is an auto-context rail assembled live from what HQ already knows happened that day — completed tasks, filed knowledge, Granola meetings, project activity — plus a placeholder for a future "Daily Spark" section (inspirational quotes, things-you-didn't-know, puzzles).

1 · Why

HQ captures what Sonja does (tasks, knowledge, meetings) but nothing captures what she thinks about the day. A journal closes that loop — and because HQ already holds the day's activity, it can do what a paper diary can't: show the day back to her while she writes. Capture friction must be near zero: type, dictate, or speak to Siri from the phone.

3 · Data model

CREATE TABLE journal_days (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id      UUID NOT NULL REFERENCES orgs(id) ON DELETE CASCADE,
  user_id     UUID NOT NULL REFERENCES user_profiles(id) ON DELETE CASCADE,
  entry_date  DATE NOT NULL,
  body        TEXT NOT NULL DEFAULT '',
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (org_id, user_id, entry_date)
);
-- updated_at trigger: handle_updated_at (standard)
-- RLS — OWNER-ONLY (OQ-1 resolved: stricter than the usual org scope,
-- because a diary is per-person even inside an org):
--   journal_days_all FOR ALL
--     USING (user_id = auth.uid() AND org_id = get_my_org_id())
--     WITH CHECK (user_id = auth.uid() AND org_id = get_my_org_id());
-- + append RPC: journal_append(p_entry_date DATE, p_chunk TEXT)
--   SECURITY INVOKER upsert → INSERT ... ON CONFLICT (org_id,user_id,entry_date)
--   DO UPDATE SET body = COALESCE(journal_days.body,'') || p_chunk
  • Auto-context is computed, never stored — assembled live from existing tables (the staleness precedent: compute, don't denormalize).
  • The first unique date-keyed pattern in HQ (the external-lineage index is the only prior unique index; nothing day-keyed exists).
  • The knowledge_entries.source CHECK gotcha is moot for v1 — journal writes never touch knowledge_entries. (Verify the live constraint anyway before any future work leans on source.)

7 · Open questions all resolved 2026-07-13

OQ-1 · Journal privacy vs org RLS

Standard org RLS means a future org teammate could read Sonja's diary. Options: (a) accept for now (single-user org today), (b) tighten the policy to user_id = auth.uid() from day one. Recommendation: (b) — it's one line in the migration, and a diary is the clearest "owner-only" case in HQ. ✅ Resolved: (b) owner-only.

OQ-2 · "Knowledge filed that day" definition

created_at only (simple), or also items filed that day (needs a filed-at timestamp we don't store — updated_at is a weak proxy)? Recommendation: created_at only in v1; a filed_at column is a cheap later add if the distinction matters. ✅ Resolved: created_at only.

OQ-3 · Dashboard chip tone

Neutral "📓 Journal" link vs a nudge ("📓 Nothing journaled today") when today is empty? Recommendation: neutral link with a subtle empty/filled state — nudges are easy to add, hard to unsee. ✅ Resolved: neutral chip.
