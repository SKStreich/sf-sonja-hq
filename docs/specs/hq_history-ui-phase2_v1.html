Why this slice

Audit v2 row 4 locked Phase 2 of the History UI work. Phase 1 (PR #30) shipped author byline + "Current" pill on the knowledge HistoryTab — the data was already in knowledge_versions. Phase 2 extends the same idea to tasks + projects, which today have no persisted history of field changes. The only thing close is the per-row updated_at timestamp, which loses every prior value.

Without history, "who moved this task to Done last week, and from what?" is unanswerable. With history, the dashboard gets a recent-activity feed and (later, out of scope here) detail pages get a timeline.

Schema

New table: entity_history

CREATE TYPE entity_history_entity_type AS ENUM ('task', 'project');

CREATE TABLE entity_history (
  id              UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  org_id          UUID NOT NULL REFERENCES orgs(id) ON DELETE CASCADE,
  entity_type     entity_history_entity_type NOT NULL,
  entity_id       UUID NOT NULL,           -- tasks.id or projects.id (no FK; polymorphic)
  field_name      TEXT NOT NULL,
  previous_value  TEXT,                    -- nullable; values cast to TEXT on write
  new_value       TEXT,                    -- nullable; values cast to TEXT on write
  changed_by      UUID REFERENCES auth.users(id),  -- nullable; null = system/migration
  changed_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_entity_history_lookup
  ON entity_history (entity_type, entity_id, changed_at DESC);
CREATE INDEX idx_entity_history_feed
  ON entity_history (org_id, changed_at DESC);
design note · polymorphic FK

entity_id is intentionally not a foreign key. Tasks and projects live in separate tables; a polymorphic FK isn't supported in Postgres. On parent delete, history rows become orphan rows — that's fine (history is append-only and survives deletes). UI joins are filtered by entity_type + a left join to the live table; deleted entities just render as "(deleted task)".

Trigger semantics

CREATE OR REPLACE FUNCTION record_task_history() RETURNS TRIGGER AS $$
DECLARE
  actor UUID := auth.uid();   -- null when called from a non-authed context
BEGIN
  IF NEW.status IS DISTINCT FROM OLD.status THEN
    INSERT INTO entity_history (org_id, entity_type, entity_id, field_name,
                                previous_value, new_value, changed_by)
    VALUES (NEW.org_id, 'task', NEW.id, 'status',
            OLD.status::text, NEW.status::text, actor);
  END IF;
  -- repeat IS DISTINCT FROM block for priority, due_date, assignee_id, project_id
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER tasks_history_capture
  AFTER UPDATE ON tasks
  FOR EACH ROW EXECUTE FUNCTION record_task_history();

-- parallel function record_project_history() + trigger projects_history_capture
  • IS DISTINCT FROM handles NULL correctly (unlike !=). No row emitted when both sides equal, including both-NULL.
  • One row per changed field per UPDATE. An UPDATE that changes status + priority emits 2 rows with identical changed_at.
  • Actor = auth.uid(). NULL when run via service-role (Server Action with elevated client) or migration — that's intentional. Server Actions that should attribute to the user must pass through the user-scoped client.
  • SECURITY DEFINER so the trigger can insert into entity_history regardless of the caller's RLS context. The trigger only writes well-formed rows derived from a row the caller already had permission to UPDATE.
  • No INSERT/DELETE coverage this slice. Task creation isn't a "change," and deletion is rare in HQ (status='cancelled' is the soft-delete). Easy follow-up if needed.

RLS policy

ALTER TABLE entity_history ENABLE ROW LEVEL SECURITY;

-- Read: tenant-scoped, matching parent table tenancy.
-- Task confidentiality is enforced by joining to the live tasks row at read time;
-- if you can't SELECT the task, your history rows for it filter out.
CREATE POLICY "entity_history_read" ON entity_history
  FOR SELECT USING (
    org_id = (SELECT org_id FROM user_profiles WHERE id = auth.uid())
    AND (
      entity_type = 'project'
      OR entity_type = 'task' AND EXISTS (
        SELECT 1 FROM tasks t
        WHERE t.id = entity_history.entity_id
      )  -- the tasks SELECT itself applies confidentiality_tier filter via tasks RLS
    )
  );

-- No INSERT policy: only the SECURITY DEFINER trigger writes. Direct INSERTs blocked.
-- No UPDATE / DELETE policy: history is append-only.

Server query shape

SELECT
  eh.id, eh.entity_type, eh.entity_id, eh.field_name,
  eh.previous_value, eh.new_value, eh.changed_at,
  eh.changed_by,
  up.full_name AS actor_name,
  COALESCE(t.title, p.name) AS entity_name,
  t.project_id AS task_project_id,
  pp.name AS task_project_name
FROM entity_history eh
LEFT JOIN user_profiles up ON up.id = eh.changed_by
LEFT JOIN tasks t ON eh.entity_type = 'task' AND t.id = eh.entity_id
LEFT JOIN projects p ON eh.entity_type = 'project' AND p.id = eh.entity_id
LEFT JOIN projects pp ON t.project_id = pp.id
ORDER BY eh.changed_at DESC
LIMIT 20;

RLS filters the result automatically. Deleted entities show entity_name = NULL → render as "(deleted task)" / "(deleted project)".

Field-aware copy rules

field_nameTemplate
status{actor} moved {entity_type} "{name}" from {prev} to {new}
priority{actor} changed priority on {entity_type} "{name}" from {prev} to {new}
due_date{actor} set due date on {entity_type} "{name}" to {new} (or "cleared due date" if new is null)
assignee_id{actor} reassigned task "{name}" to {new_user} (or "(unassigned)")
project_id{actor} moved task "{name}" to project {new_project}
phase{actor} set phase on project "{name}" to {new}

Actor "(System)" when changed_by is NULL.

Open questions for review

  1. Feed position on dashboard. Above or below the existing "Recent activity" (project_updates) section? Or merged into one combined "Recent" feed?
  2. Feed length. Top 20 rows? Top 10? Pagination/"show more" or hard cap?
  3. Backfill on migration. Skip (history starts now, prior state lost) or seed one row per existing row with current values? Recommendation: skip — backfill adds noise, current state is already visible elsewhere.
  4. Service-role attribution. Some Server Actions use the service-role client (bypasses RLS). Those changes will land with changed_by = NULL → "(System)" in the feed. Acceptable, or should we thread the user_id through Server Actions so they attribute correctly? Recommendation: acceptable for v1.
  5. Confidentiality on task history. Current RLS proposal: history readable iff parent task readable. Means a task that goes confidentiality_tier = 'private' hides its history from teammates immediately. Sound right?

Out of scope (this slice)

  • Detail-page timelines on /dashboard/tasks/[id] and /dashboard/projects/[id]. Per intake Q2 you picked dashboard feed only. Trivial follow-up once the data exists.
  • Other tracked fields (title, description, tags). Trigger easily extended.
  • INSERT / DELETE events. Only UPDATE changes captured.
  • Merge with project_updates. Out of scope — that's manual update content, not field-change history. Could merge feeds in UI later.
  • Filtering / search on the feed. Top-N reverse-chronological only.

Test plan

  • Trigger correctness (real DB per feedback-testing):
    • UPDATE that changes status emits exactly 1 row with correct prev/new/actor.
    • UPDATE that changes 3 tracked fields emits 3 rows, same changed_at.
    • UPDATE that touches only updated_at (no tracked field) emits 0 rows.
    • UPDATE NULL → value (and value → NULL) emits a row.
    • UPDATE NULL → NULL emits 0 rows.
  • RLS isolation:
    • User in org A cannot read history rows for org B.
    • User cannot read history for a private task that isn't theirs.
    • Direct INSERT into entity_history by authenticated user is rejected (no INSERT policy).
  • ActivityFeed component:
    • Renders status/priority/due_date/assignee/project/phase rows with the right templates.
    • Renders "(deleted task)" when join misses.
    • Renders "(System)" when actor missing.
