1 · The layer model — what lives where, what backs it up

Five layers. Each has a primary provider and a defined restore path. The rule of thumb: every layer is recoverable from a provider independent of the one that failed.

LayerHoldsPrimary providerIf it fails / corrupts → restore from
1CodeAll source + full git historyGitHub (canonical)Every local clone has full history; re-push. Optional: add a 2nd git mirror for provider redundancy.
2HostingRunning apps / deploysVercel (apps) · Netlify (static)Redeploy from GitHub in minutes. Stateless — nothing unique lives here.
3DataDatabases + uploaded filesSupabase (Postgres + Storage)Supabase PITR (paid tier) + Layer 4 off-platform dumps.
4Off-platform DRDaily DB dumps + storage-bucket copiesCloudflare R2 (independent of Supabase/Vercel/GitHub)This IS the restore source for Layer 3. Must be rehearsed (see open actions).
5Local + personal backupWorking copies on disk; personal documentsLocal disk · OneDrive (documents only)Working copies re-clone from GitHub. OneDrive is NOT the repo backup — see §3.
The honest gap

Layer-4 backups exist and run, but a backup you've never restored from isn't fully proven. A monthly health-check rehearsal (row 16 tier-1) now verifies dumps are fresh / intact / restorable, but a full restore into a live DB (tier-2) is still un-exercised. That's the remaining backup risk — not which folder the code sits in.

3 · Locked decisions

D1 · HQ repo relocated out of OneDrive DONE 2026-05-31

The HQ working copy moved from ~/OneDrive/Documents/GitHub/HQ/sf-sonja-hq to ~/Projects/sf-sonja-hq. Fresh clone from GitHub (verified: main, clean, file count matches, git fsck clean); OneDrive copy deleted; stale ~/dev/ worktrees orphaned.

D2 · The backup model is GitHub + R2 — not OneDrive

  • Code is backed up by GitHub (+ every local clone). OneDrive duplicating the working tree added nothing GitHub didn't already cover.
  • Data (DB + uploaded files) — the genuinely irreplaceable layer — is backed up daily to Cloudflare R2, an independent provider. OneDrive never covered this (it's not in the repo).
  • OneDrive's role is personal/document backup only. Syncing git repos through it caused deleted files to resurrect and made local tooling (tsc/vitest/next dev, even grep) slow via on-demand round-trips. Repos do not belong in OneDrive.

D3 · Repo relocations — ALL repos out of OneDrive DONE 2026-06-01

SF Ops, SF Solutions, and SF Website all relocated out of OneDrive to ~/Projects/ (same pattern as HQ; OneDrive local copies removed). Each verified: correct main HEAD, git fsck clean, all working-tree files (incl. untracked + .env files) preserved, node_modules reinstalled. OneDrive's GitHub/SF/ folder is now repo-free — every SF platform repo (HQ, Ops, Solutions, Website) lives under ~/Projects/, matching D2.

Relocation gotcha — for any future repo move out of OneDrive

A plain mv of a repo across the OneDrive file-provider boundary times out / thrashes on large node_modules (sf-ops ballooned mid-move and had to be killed). Pattern that worked for both Ops and Website: delete node_modules first (it's regenerable), move the small repo, then npm install in the new location. Verify git fsck + HEAD + untracked files after.
