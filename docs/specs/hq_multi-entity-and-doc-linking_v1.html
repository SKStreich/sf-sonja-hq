1 · Where we are today

Entity ("personality") is single-valued everywhere

RecordColumnShape
Knowledge entryknowledge_entries.entityTEXT NOT NULL — one of tm·sf·sfe·personal (CHECK enum)
Projectprojects.entity_idUUID NOT NULL → FK to entities table
Tasktasks.entity_idUUID NOT NULL → FK to entities table

The two record types represent entity differently: knowledge uses a text enum, projects/tasks use a FK to the entities table (one row per (org, type)). The junction design respects each parent's native representation to keep churn low.

Key fact that makes this cheap

Entity is a label / filter dimension, not a security boundary. RLS on projects, tasks, and knowledge is keyed on org_id (plus task confidentiality tier) — never on entity. So moving to many-to-many touches display and filtering, not the security model. The junction tables only need standard tenant-isolation RLS.

Document → project/task linking: the rails exist, the train doesn't

knowledge_links already carries three target columns — to_entry, to_project, to_task — under an XOR constraint (exactly one target per row, many rows per document allowed). So a document can already point at many projects/tasks at the data layer.

But every such row today is written by one mechanism only: the workspace-page [[…]] mention scanner (relation='mentions') in src/app/api/knowledge/links.ts. There is no deliberate "Attach this document to this project" action, and project/task pages surface only incidental mentions via getProjectBacklinks() — not curated attachments.

2 · Target model — Request 2 (multi-entity)

Two junction tables, each mirroring its parent's entity representation:

-- knowledge entries use the text enum, so the junction does too
CREATE TABLE knowledge_entry_entities (
  entry_id    UUID NOT NULL REFERENCES knowledge_entries(id) ON DELETE CASCADE,
  entity      TEXT NOT NULL CHECK (entity IN ('tm','sf','sfe','personal')),
  org_id      UUID NOT NULL REFERENCES orgs(id) ON DELETE CASCADE,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (entry_id, entity)
);

-- projects use the entities FK, so the junction does too
CREATE TABLE project_entities (
  project_id  UUID NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  entity_id   UUID NOT NULL REFERENCES entities(id) ON DELETE CASCADE,
  org_id      UUID NOT NULL REFERENCES orgs(id) ON DELETE CASCADE,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (project_id, entity_id)
);

-- tenant isolation RLS (entity is NOT a security boundary; org_id is)
ALTER TABLE knowledge_entry_entities ENABLE ROW LEVEL SECURITY;
ALTER TABLE project_entities         ENABLE ROW LEVEL SECURITY;
-- policy: org_id = get_my_org_id()  (mirrors every other HQ table)

Backfill — every existing row gets exactly one junction row

INSERT INTO knowledge_entry_entities (entry_id, entity, org_id)
  SELECT id, entity, org_id FROM knowledge_entries;

INSERT INTO project_entities (project_id, entity_id, org_id)
  SELECT id, entity_id, org_id FROM projects;

After backfill, the app reads entity membership from the junctions. The old single columns (knowledge_entries.entity, projects.entity_id) — whether we drop them or keep them as deprecated — is open question OQ1 below.

"At least one entity" — the flat model's one sharp edge

Today a NOT NULL column guarantees every record has an entity. A junction table can't express that with a simple constraint — a record could end up with zero rows. We must enforce "≥ 1 entity" somewhere. See OQ2.

3 · Target model — Request 1 (document linking)

Reuse knowledge_links with a new, deliberate relation value 'attached' (distinct from the auto-generated 'mentions'). One row per attachment: 

-- extend the existing relation CHECK to add 'attached'
ALTER TABLE knowledge_links DROP CONSTRAINT knowledge_links_relation_check;
ALTER TABLE knowledge_links ADD CONSTRAINT knowledge_links_relation_check
  CHECK (relation IN ('cites','duplicate_of','extends','chat_about',
    'merged_into','critique_of','note_on','superseded_by','mentions','attached'));

-- an attachment is: from_entry (the document) → to_project OR to_task
-- row example:
--   from_entry = <doc id>, to_project = <project id>, relation = 'attached'

Two new server actions — attachEntryTo(entryId, {projectId | taskId}) and detachEntry(linkId) — write/delete these rows explicitly (not via the mention scanner). The XOR target constraint already guarantees a row points at exactly one of project/task. Linking targets both projects and tasks, per Request 1.
