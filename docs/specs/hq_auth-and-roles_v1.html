Hidden decision — existing HQ users have no password

Sonja and any current collaborators authenticated only via magic link. There are no auth.users.encrypted_password rows for them. Four options to onboard them onto password auth:

OptionHow it worksProsCons
1A-A REC First failed sign-in (no password) shows "Set your password" CTA → magic-link → set-password page Self-serve · no admin work · familiar magic-link flow · zero deploy-day side effects Confusing first-time error message ("invalid login") unless we detect "no password set" and reframe
1A-B Bulk email blast on deploy with "Set your password" magic links to all current users Proactive · no error state · users prepared Email blast feels heavy for ~handful of users · still need per-user link generation script · still need the set-password page
1A-C Sonja (as owner) sets passwords manually for each user via Supabase admin Zero new code paths · simple Doesn't scale past 5–10 users · Sonja has to communicate the password OOB · still need set-password page for in-app rotation
1A-D Add "Set Password" section in profile page; users sign in via magic link first, then set password there No error state · explicit user action Multi-step (sign in → navigate to profile → set) · users may not discover · password becomes optional indefinitely
Recommendation
1A-A. Detect "user exists but no password" by querying auth.users via service-role check after a failed sign-in, and reframe the error as "This account uses magic-link login. Click here to set a password." Same set-password page is reusable for invite + forgot-password flows. One code path, three entry points.

OldNewSemantic changeData migration

Migration shape

-- migration: 20260524000001_canonical_role_spine.sql
BEGIN;

-- Postgres 15+ supports RENAME VALUE inside a transaction since 13.
ALTER TYPE user_role RENAME VALUE 'owner' TO 'platform_owner';
ALTER TYPE user_role RENAME VALUE 'admin' TO 'org_admin';

-- ADD VALUE must be outside a transaction in older PG; PG 12+ relaxed this.
ALTER TYPE user_role ADD VALUE 'supervisor' AFTER 'org_admin';

-- RLS policies that hard-coded old names
DROP POLICY "integrations_admin_only" ON integrations;
CREATE POLICY "integrations_admin_only" ON integrations FOR ALL
  USING (
    org_id = (SELECT org_id FROM user_profiles WHERE id = auth.uid())
    AND EXISTS (
      SELECT 1 FROM user_profiles
      WHERE id = auth.uid()
        AND role IN ('platform_owner', 'org_admin')
    )
  );

-- Same shape for the 2 policies in 20260425…contacts_and_share_forwarding.sql

COMMIT;

Note: ALTER TYPE … RENAME VALUE updates the enum in place — no data rewrite, all existing rows automatically reflect the new label. RLS policies need explicit drop+recreate because Postgres caches the policy text against the old enum string.

Open decisions for Action 5

5A · Semantic for the new supervisor role in HQ

OptionWhat it means in HQEffect
5A-1 REC No HQ semantic — exists in the enum for cross-platform consistency only. Not surfaced in the invite dropdown. No callsite treats it specially. Lowest risk · matches audit's "canonical spine for both projects" framing · easy to add semantic later
5A-2 Between org_admin and member — can manage projects/tasks/knowledge but not org settings/billing Adds a real permission tier · requires defining the cutline + updating RLS + UI gates · higher scope
5A-3 Defer entirely — don't add supervisor to the HQ enum yet HQ enum becomes 4 values (renamed but no add) · diverges from the "canonical spine" goal of action 5

OptionEffect

5B-2 No — only expose org_admin · member · read_only until 5A is decided

3 · Implementation sequencing

Two PRs after this spec is signed off:

Why Action 5 first
Role rename touches user_profiles which is queried in the auth callback path. Doing role rename first means Action 1's auth code sees a stable schema. Reverse order would mean Action 5 has to touch Action 1's new files too.

4 · What's NOT in this spec

  • SF Ops role changes. Sonja's call: HQ-only migration · SF Ops keeps platform_owner · org_admin · supervisor · field_tech · contractor indefinitely.
  • Brand-styling the login page. Lives in Action 2 (separate PR, runs in parallel with Sonja reviewing this spec).
  • 2FA / TOTP / passkeys. Out of scope for the alignment sprint. Revisit in a future security-themed sprint.
  • Re-defining what each role can do. This spec preserves existing permission boundaries; it only renames owner→platform_owner and admin→org_admin. If 5A-2 wins, a follow-up spec defines the supervisor cutline.
  • Cross-org access (true platform-owner across multiple orgs). HQ today is effectively single-org for Sonja. platform_owner is a rename, not a new capability.
