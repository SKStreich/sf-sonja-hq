HQ Databases — a Notion-parity primitive

Version 1 (draft) · 2026-06-20 · Owner: Sonja Streich · Status: OPEN — awaiting lock

North star: retire Notion. This spec scopes the one capability HQ is missing to get there — structured, tabular data (Notion "databases"), not just documents.

1 · Why this exists

HQ today stores documents: a knowledge entry is title + body + tags + entities, optionally rendered as an HTML snapshot in an "Original" tab. That is an archive. It cannot represent the thing that carries most of the real substance in a serious Notion workspace: a database — a collection of records with typed columns, select options, relations, and views.

The trigger. The migrated Semantic Layer — Metric/KPI Definitions page came across empty of its tables. Root cause: its content is two Notion databases (App Sections, Metrics & Calculations), and the migrator was fed a markdown export — Notion databases don't serialize to markdown, so they appeared as [Embedded database: …] placeholders. No rows were ever in the file. This is true of every database across the workspace, not one page.
The data is recoverable. Verified live via the Notion API: the Metrics & Calculations database has a full 14-property typed schema and every row is fetchable (each row is a child page). Example row Traded Unit Ratio returned its Backend Variable Name (traded_unit_ratio_past28days), Calculation Summary, Implementation Status (✅ Implemented), Time Period (MTD), Viz Type (Ratio), and Source Ticket (BPE-903). Nothing is locked away.

2 · Two tracks (decided 2026-06-20: do both)

TrackWhatDeliversLifespan
A · Stopgap Importer pulls DB rows via the Notion API and renders them as static HTML tables inside the existing snapshot entries. Lost content (Semantic Layer metrics + every other embedded table) becomes visible again — fast. Throwaway. Read-only. Superseded by Track B.
B · The primitive A real Databases feature in HQ: typed records, table view, inline-on-page embedding, a proper Notion-API importer. A living tool you can edit in — the thing that actually lets Notion be retired. Permanent. This spec is about Track B.

Track A is intentionally not over-built — it exists so nothing is missing while Track B is designed and shipped. Its HTML tables get replaced by real embedded databases once Track B lands.

3 · Scope of Track B v1 (proposed)

In

  • Database object: a named collection belonging to an org + entity set (reuses the multi-entity model).
  • Typed properties: text, number, select, multi_select, status, checkbox, date, url, and relation (to another HQ database row).
  • Records with values per property; one designated title property.
  • Table view: sortable columns, column show/hide, basic filtering by property.
  • Inline-on-page: embed a database (or a filtered view of it) inside a knowledge/workspace page — the Notion "page with a table in it" shape.
  • Notion-API importer: given a Notion database URL, recreate its schema + all rows in HQ (replaces the markdown migrator for anything tabular).
  • Editing: add/edit/delete rows and columns in-app.

Out (deferred — Phase 2+)

  • Board / calendar / timeline / gallery views.
  • Formulas, rollups, and aggregation footers.
  • Two-way relation sync & rollup recompute.
  • Real-time multiplayer editing; per-cell comments.
  • Permissions below the row level (entity tagging + org RLS is the boundary, per the existing model).

4 · Data model (proposed)

Three core tables, all org-scoped under the existing RLS model. Entity tagging reuses the junction approach from the multi-entity work.

hq_databases
  id            uuid pk
  org_id        uuid  -- RLS boundary (reuse existing pattern)
  title         text
  icon          text null
  description   text null
  created_by    uuid
  created_at / updated_at

hq_db_properties              -- the schema (columns)
  id            uuid pk
  database_id   uuid fk -> hq_databases (on delete cascade)
  name          text
  type          text  -- text|number|select|multi_select|status|checkbox|date|url|relation
  position      int
  config        jsonb -- select options[], relation target db id, number format, etc.
  is_title      bool  -- exactly one true per database

hq_db_records                 -- the rows
  id            uuid pk
  database_id   uuid fk -> hq_databases (on delete cascade)
  position      int
  values        jsonb -- { "<property_id>": <value> }  (typed at app layer)
  created_at / updated_at

hq_db_entities (junction)     -- entity tags, mirrors knowledge_entry_entities
  database_id   uuid
  entity        text
Why JSONB values, not a column-per-property table? Notion-style schemas are user-defined and change often. A normalized record_values(record_id, property_id, value) table is the alternative (cleaner querying, easier single-property indexes). The JSONB approach is simpler to build and read whole-row, at the cost of harder server-side filtering on a single property. Recommendation: JSONB for v1 (filtering/sorting volumes are small; revisit if a database grows past a few thousand rows). Flagged as OQ-2.

Inline embedding

A page embeds a database via a marker the renderer expands — e.g. a row in a new knowledge_db_embeds(entry_id, database_id, view_config) table, or a [[db:<id>]] token in the body resolved at render time. The first is cleaner for filtered/sorted per-embed views. Flagged as OQ-3.

5 · The Notion-API importer (replaces the markdown migrator for tables)

  1. Input: a Notion database URL (or a page URL — auto-discover databases beneath it).
  2. Fetch the database → map each Notion property type to an HQ property type (table below).
  3. Enumerate row-pages → read each row's properties → insert hq_db_records.
  4. Resolve relation columns in a second pass once both databases exist (store Notion-id → HQ-id map).
  5. Report what didn't map (e.g. formulas/rollups → imported as a read-only text snapshot of their current value).
Notion typeHQ typeNote
titletext (is_title)exactly one
rich_texttextstrip/keep inline formatting → markdown
select / statusselect / statuscarry option names + colors
multi_selectmulti_selectcarry options
number / checkbox / date / urlsamedirect
relationrelation2-pass id remap; only if the target db is also imported
formula / rolluptext (read-only snapshot)v1 can't recompute — store current value, flag it

Auth: the connected Notion MCP works token-free for reads in-session (verified). A standing/automated importer would use a Notion internal-integration token in HQ env (mirrors the existing API-key pattern). Flagged as OQ-4.

6 · UI surface (proposed)

  • New "Databases" tab in the Knowledge hub (peer of Cards / List / Pages / Insights / Vault).
  • A database opens to its table view; click a row → a record panel (all properties, editable).
  • "+ New database" (blank or "Import from Notion"); "+ New row"; "+ New column" (pick type + options).
  • On a page: an "Embed database" block; renders the live table inline.

8 · Open questions to lock

OQ-1 · How much of Notion is "enough to retire Notion"?

Is v1's table-view + import + editing sufficient to move off Notion for your real pages, or is a specific second view (calendar? board?) a hard requirement before you'd switch? Recommendation: ship table-view first, add views by demand.

OQ-2 · Storage shape

JSONB values (simpler) vs normalized record_values (better single-property querying). Recommendation: JSONB for v1.

OQ-3 · Inline-embed mechanism

Embed table (per-embed filtered views) vs [[db:id]] body token. Recommendation: embed table.

OQ-4 · Importer auth

In-session MCP (manual, token-free) for the first imports vs a stored Notion integration token for repeatable/automated import. Recommendation: MCP-manual for B2; add a token only if you want re-syncs.

OQ-5 · Re-sync vs one-time

Is the Notion import a one-time migration (then HQ is the source of truth) or do you want ongoing Notion→HQ re-sync during a transition? Recommendation: one-time; HQ becomes source of truth (matches the north star).

OQ-6 · Where do the imported databases live?

Same tm entity as the 21 doc migration, or a different tag? Do the metric databases belong under a TM/CTHQ entity?
