✅ What worked

Patterns that earned their keep. Carry these forward.

Single-table data models over many specialized ones

Sprint 6 collapsed 4 tables into knowledge_entries + supporting tables. Filter by kind instead of multiplying schemas. Shipping new kinds (chat, critique, workspace) became a one-line constraint change.

Server actions for everything

No REST routes, no API design discussions. Every mutation is a typed RPC. Easy to test, easy to reason about, plays well with React's startTransition for optimistic UI.

Sandboxed iframes for HTML rendering

User uploads can include arbitrary HTML/CSS. <iframe srcDoc sandbox=""> isolates them and lets us style externally. Used for the Original viewer and share viewer.

Admin client for public flows

Public /share/[token] route uses the admin client to bypass RLS. The auth-gated and public surfaces stay cleanly separated. Same pattern works for Cron.

PR-driven prod deploys via gh CLI

Every change ships through gh pr create → wait for green → gh pr merge. Auto-deploys. Rollback is one click on Vercel. Velocity stayed high because the loop was fast.

Memory + roadmap files in ~/.claude

Across 15+ sessions, the roadmap and tech docs stayed in sync because every session ended with a memory update. Future me always knew where to pick up.

Iterative slicing (10a → 10b → 10c)

Big features got sliced into shippable chunks. Workspace pages went out as Slice 1 (hierarchy + Markdown) before backlinks/slash-commands were even designed. Avoided mega-PRs.

Browser preview testing alongside unit tests

Unit tests caught logic bugs (78/78 stayed green). Browser preview caught the things tests miss: dark-mode CSS bugs, prose blocking checkboxes, race conditions in optimistic UI.

Additive migrations only

Every schema change was ADD COLUMN IF NOT EXISTS or new table. No destructive ALTERs. Combined with Supabase PITR, rollback risk stayed low.

Two-track auth (user + admin client)

User context drives RLS for normal flows; admin client bypasses RLS for cron, public viewers, and system inserts. Two clear paths instead of one fuzzy one.

⚠️ What didn't work the first time

Patterns we landed on after a false start. Worth flagging because the false start is the natural default.

Silent try / catch on external API calls

Resend errors were swallowed for weeks. We had to ship PR #4 explicitly to surface them. Default for any external call: log the error, don't return success.

Relying on router.refresh() alone for list mutations

Bit us 3 times — task add, project file upload, share creation. Server re-rendered but local useState didn't. Always optimistically update local state before calling refresh.

CHECK constraints rejecting silently

Inserting 'share_forward_request' into notifications.type failed because it wasn't in the CHECK enum. Bell stayed empty for weeks. Always await + check error on every insert.

MCP apply_migration ≠ committed migration

Migrations applied via the Supabase MCP only exist in the dev DB. Almost shipped PR #2 with 4 missing local files. Hand-write the .sql file every time.

Z-index inside fixed nav stacking context

Notification dropdown's "click outside to close" backdrop was trapped inside the nav's z-40 layer, so clicking the rest of the page didn't dismiss. Use document-level mousedown listeners for outside-click, not z-index tricks.

Ambiguous PostgREST embeds

Two FKs from share_forwarding_requests to knowledge_shares made !inner embeds explode with "more than one relationship found". Always specify the FK constraint name when a table has multiple FKs to the same target.

Webpack cache breaks on new directories

Added a new views/ folder → CSS broke silently or webpack-runtime crashed. Standard fix: rm -rf .next && restart dev. Document this in the gotchas file.

Tailwind prose styles conflict with form controls

Native checkboxes inside a prose-styled preview didn't fire onChange — the styling killed pointer events. For interactive elements inside prose, render a styled button, not a native input.

Server actions can't accept plain objects with File

"Only plain objects can be passed" error. Had to convert to FormData. Any action that takes a File needs a FormData signature.

Chrome Auto Dark Mode inverted explicit colors

Login screen looked dark even though every Tailwind class was light. Browser was guessing. Always declare :root { color-scheme: light } in globals.css from day 1.

🎯 Hard lessons (gotchas that cost us hours)

Specific land mines. Each one ate at least an hour of debug time. Each one has a one-line fix.

1

Resend domain verification + API key match are pre-requisites for ANY transactional email

The Resend domain (streichforce.com) needed verified DNS records (SPF + DKIM) before any send would deliver. Plus the API key in Vercel must match the team that owns the domain — not just any key from the dashboard.

Carry-over: For SF Ops, set up Resend domain verification + production env var on the day you create the project. Don't wait until the first email feature ships.
2

Supabase auth Redirect URL allow-list must include preview deployments

Magic links from a Vercel preview URL bounced back to hq.streichforce.com because the preview URL wasn't whitelisted. Result: you could never test login on a preview build.

Carry-over: When provisioning the SF Ops Supabase project, add the wildcard https://*-streich-force.vercel.app/auth/callback to the allow-list immediately.
3

Local next build catches type errors faster than Vercel does

PR #6 first deploy failed because a refactor missed a default-fields object literal. The error was a clean TypeScript message, but we burned a Vercel cycle to see it.

Carry-over: Run npx next build locally before git push on any PR that touches types. Faster than waiting for Vercel.
4

Vercel Cron requires CRON_SECRET + redeploy after env var add

Adding the env var doesn't apply to existing deployments. The cron route stayed in the "503 cron not configured" state until we redeployed.

Carry-over: Build the env-var checklist before merging any PR that introduces a cron route. Document the redeploy step in the PR body.
5

Adding new entity_type enum value requires two migrations

Postgres requires ALTER TYPE … ADD VALUE to commit before the value can be used in the same transaction. Adding 'sfc' + inserting the SF-Containers entity row had to be split.

Carry-over: Plan two migration files whenever you extend an enum and immediately reference the new value in a row insert.
6

Optimistic UI is the default, not a nice-to-have

"It saved but didn't appear" was reported three separate times for three different list mutations. Each time the fix was the same: have the server action return the inserted row, push it into local state, then call router.refresh() for the side effects.

Carry-over: Standard pattern for any list-mutating action: return the row, optimistically insert into client state, then refresh. Document it in the SF Ops dev guide.
7

"Notification rang" ≠ "notification was created"

The bell looked broken because submitForwardRequest's notification insert silently failed against a CHECK constraint. await without checking error meant the function returned success.

Carry-over: SF Ops will have many notification types (work-order assignments, status changes, comments). Bake the type allow-list policy into the migration template and treat every insert/upsert as .then(({ error }) => if (error) log + handle).
8

The default `prose` typography plugin disables interactive form elements

GFM task list checkboxes wouldn't toggle inside a Markdown preview. The @tailwindcss/typography plugin's default styles set pointer-events: none on inputs.

Carry-over: If the work-orders module ever renders user-editable Markdown (likely — comments, descriptions), wrap interactive elements in custom components, not native inputs.
9

Webpack/.next cache invalidation is silent and routine

New folders, new exports, sometimes just a renamed component → dev server starts emitting "Cannot read properties of undefined (reading 'call')" or unstyled HTML. Nothing in the error logs hints at the cause.

Carry-over: Whenever a dev session "feels broken," kill server && rm -rf .next && npm run dev first, debug second. Add this to the team onboarding doc.
10

Multi-FK PostgREST joins need explicit names

When two FKs go from table A to table B, a PostgREST embed like knowledge_shares!inner(...) blows up with "more than one relationship found". You have to write knowledge_shares!share_forwarding_requests_share_id_fkey!inner(...).

Carry-over: Work orders likely link to multiple tables (assignee, customer, asset, parent_order). Plan named FKs from the start: fk_work_orders_assigned_to_user, fk_work_orders_customer, etc.
11

The 30-minute 2FA pass should happen before the first prod deploy, not after

We documented and recommended the DR Minimum 2FA checklist after PR-A. Doing it first avoids a window where the system is live but the credentials defending it are single-factor.

Carry-over: Day 1 of SF Ops, before deploying anything: enable 2FA on Supabase, Vercel, GitHub, Cloudflare DNS, Resend, Anthropic, and your email. Save recovery codes to a password manager.
12

Storage backups are the unaddressed gap

Free-tier Supabase has no off-platform storage replication. If the Supabase project gets locked or deleted, every uploaded file is gone. We documented this as the "DR Standard" upgrade path but didn't ship it.

Carry-over: SF Ops will store work-order attachments, photos, signed PDFs — content that's legally important to retain. Plan an R2 nightly sync from day 1, or accept the risk explicitly.

🛠 Specific notes for the work-orders module

You mentioned you're already building this. Here's what transferable from Sonja HQ would land cleanest in that domain.

Schema patterns to reuse

Single status enum + filter pills

Tasks landed on open / done / cancelled. Work orders likely have more (draft, scheduled, in_progress, on_hold, complete, billed, voided). Same UX pattern: count tiles double as filters with an active ring.

parent_id for nesting

Workspace pages got hierarchy via parent_id. Work orders may have parent/child for change orders or split jobs. Same column pattern, with a cycle check on move.

attachments via Supabase Storage + metadata table

Knowledge Hub uses storage_path + mime_type on the entry row plus signed-URL access. Work orders probably need the same: photos, signed PDFs, customer-supplied files.

audit log via versions table

We have knowledge_versions for every saved revision. Work orders need the equivalent for compliance — a work_order_versions table writing on every save.

linked records via a generic links table

knowledge_links joins entries to entries with a relation enum. Work orders link to customers, assets, invoices, parent orders — a single work_order_links table beats five FK columns.

notifications with type allow-list + entity_type allow-list

Both check constraints bit us. Plan the work-order notification types upfront: wo_assigned, wo_status_changed, wo_comment, wo_due_soon, wo_completed. Migration includes them all from day 1.

Patterns to avoid

Don't model assignee, requestor, completed_by as the same FK column

If you need three relationships to users, give them three named FKs. PostgREST will choke on embeds otherwise. Lesson #10 above.

Don't store the work-order number in the URL

Use the UUID id. Numbers are display-only. They will be re-numbered in audits and migrations.

Don't trust router.refresh() alone after status change

When a tech marks a WO complete in the field, the local list must update immediately. Same optimistic-UI pattern Sonja HQ landed on.

Don't gate the public customer view (signature page, etc.) with the user's RLS

Use the admin client + a token, like /share/[token] in Sonja HQ. Customers don't have user accounts — they have signed links.

🎓 Meta-lessons (for the way we work, not the code)

Things that made the collaboration faster the second time around.

Resolve scope questions BEFORE coding

Sprint 10b kicked off with four open questions answered explicitly (watermarking, editor framework, contact ownership, backfill). Saved a half-day of back-and-forth mid-build.

Browser-verify after every UI change

When a fix involved CSS or interactive elements, the user often spotted something the unit tests missed. Make the loop one-screenshot-and-iterate.

Capture decisions in memory, not in chat

The roadmap file's "open questions resolved" section was the cleanest way to make sure we agreed on direction before writing migrations.

Phase your PRs (A → B → C)

PR-A: foundations + DB + UI scaffolds. PR-B: live data + cron + integrations. PR-C: docs + DR. Each PR is verifiable on its own. No mega-PRs.

Keep a scratch docs/ folder for reasoning, not just final docs

The 2FA checklist, retrospective, decision log — these earn their keep when you (or a future contributor) need to understand why something is the way it is.

Don't hide the build process from yourself

When the user said "I'm not a programmer," the editor toolbar got built. Doing the syntax for the user instead of asking them to learn Markdown was the right call. Same energy applies to SF Ops's field workers, customers, and admins.
